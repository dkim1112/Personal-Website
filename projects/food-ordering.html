Overview

A mobile-first web platform that handles the full lifecycle of a small recurring private event — invitation, gender-balanced registration, payment, admin approval, and real-time roster updates — built for a specific bar in Seoul and used roughly twice a week.

Yeonrim is a small bar in Seoul that hosts private matchmaking parties — guests pay to attend an evening designed around meeting other people, with a gender-balanced roster and an admin-curated guest list. The bar was running this entirely manually: invitations through KakaoTalk groups, payment tracking in a notebook, gender balance enforced by counting messages, and questionnaire responses collected on paper. Kyareureuk Party is the platform that replaced all of that. It handles invitation-only access (single shared bar password), Kakao OAuth onboarding, a structured questionnaire for participant context, gender-quota tracking with real-time roster updates via Firestore, a manual bank-transfer payment workflow with admin verification, and an admin dashboard for the bar to approve or reject registrants. The platform has been actively used for roughly two events per week, averaging ~10 successful registrations per session.

The Problem

Yeonrim was running its events entirely manually — KakaoTalk groups, paper questionnaires, gender balance enforced by counting messages. It didn't scale, and it made the staff the bottleneck.

The bar's matchmaking events have a few hard constraints that make manual coordination brittle. Gender quotas need to be enforced in real time, because the event's premise depends on roughly balanced attendance. Payment needs to be confirmed before the spot is held, but bank-transfer reconciliation by hand is slow. The bar's admin has to approve registrants individually (the events depend on trust), but that approval step needs context — who is this person, what did they say in the questionnaire — that was sitting in different chat threads. Before this platform, the bar staff was effectively a single-threaded human queue handling all of it. The platform turns that into something the bar can run on the side while doing everything else they do.

Product Vision

A platform a single small business can operate without engineering support — one bar admin, mobile-first because every guest is on their phone, opinionated about the workflow so the staff doesn't have to make process decisions each event.

The platform's actual users are two distinct populations with very different needs. The bar admin needs an at-a-glance view of who's registered, who's paid, who's been approved, and what the current gender balance is — and a one-click action to approve or reject each registrant. Guests need a mobile-first flow that's fast and doesn't require account creation friction (Kakao OAuth covers this in one tap). Both sides need real-time state so the admin's approval shows up on the guest's screen immediately. The system has handled roughly two events per week since launch, averaging ~10 successful registrations per session — small numbers, but real users on real events.

Core architecture

Mobile-first Next.js App with Firebase Real-time State and Admin-Approval Workflow

The frontend is a Next.js 16 App Router application written in TypeScript, mobile-first via Tailwind, with Radix UI primitives for accessibility on the components that matter (forms, dialogs, status indicators). The backend layer is Firebase Authentication + Firestore for storage, with Firestore security rules enforcing access control. The data flow is event-driven through Firestore real-time listeners: when the admin approves a registrant, the guest's status updates immediately on their device without a refresh. Authentication is Kakao OAuth (the dominant social login in Korea), handled through Next.js API routes for the OAuth code exchange. The payment workflow is manual bank-transfer with admin verification — initially a TossPayments integration was attempted, but it added more complexity than the bar needed for the volume, and the manual flow with admin verification turned out to be the right fit. Route protection enforces flow ordering (you have to complete the questionnaire before the payment page, etc.) through session-based state checks, not bypassable through URL manipulation.

My Contribution

Sole full-stack engineer — designed the workflow with the bar, built the entire platform end-to-end, and iterated based on what actually worked at live events.

This was a one-person build from sketches (the ideation jpeg is still in the repo) through deployed product. The most useful constraint was knowing the actual users: one admin, ~10 guests per event, mobile-only. That ruled out a lot of architectural over-engineering and pointed me toward decisions that fit the scale — Firebase over a custom backend, manual bank transfer over a payment processor, Radix primitives over a custom design system. Most of the engineering work was getting the small details right (race conditions on quota checks, OAuth state management, admin-approval visibility), not building something architecturally novel.

What I worked on

  • Full-stack architecture. Next.js App Router structure, Firebase setup, Firestore data modeling for participant state (registered → approved → paid → confirmed) and event quota tracking, security rules to restrict admin endpoints.
  • Auth and flow protection. Kakao OAuth integration via Next.js API routes, session-based state management, route guards that enforce flow ordering so guests can't skip past required steps by editing URLs.
  • Real-time admin dashboard. Live participant list with status indicators, one-click approval/rejection actions, gender-quota counter that updates as registrations come in, payment-verification interface for the manual bank-transfer flow.
  • Payment workflow iteration. Started with TossPayments (Korea's standard payment SDK), pivoted to manual bank transfer with admin verification after the integration complexity outweighed the benefit for the bar's actual volume. The pivot itself was the lesson — knowing when to drop a "more sophisticated" solution that fits the use case worse.

Lessons Learned

The right architecture for a 10-user system is not a smaller version of the right architecture for a 10,000-user system. Picking simpler tools (Firebase over a custom backend, manual bank transfer over TossPayments) wasn't a compromise — it was the correct fit.

What worked: committing to mobile-first from the very first sketch. Every guest is on their phone — there's no "desktop user" segment to consider — and designing the layouts mobile-first meant I never had to retrofit responsiveness later. Tailwind + Radix made this fast.

What didn't: the initial TossPayments integration. I assumed the bar would want a real payment processor; the bar actually wanted less infrastructure to manage, not more. The integration was building toward a complexity level the use case didn't justify. I rolled it back and built the manual bank-transfer workflow with admin verification, which fit the scale better and shipped sooner.

What I'd do differently: I'd validate the workflow with the bar admin in a paper prototype or low-fidelity Figma before writing the auth code. Several iterations on the admin dashboard happened only because I'd built features the admin didn't actually use the way I'd expected. Walking through the user flow with them before any code would have caught those misalignments earlier.

Where It's Going

The platform is in stable use by the bar with current features sufficient for their workload. Two open extensions if the bar's needs grow:

  • Automated payment verification via TossPayments callback. The integration I previously rolled back, now justified if event volume grows.
  • Returning-guest profile system. So participants don't have to re-enter the questionnaire for every event.

The honest answer is this platform is unlikely to scale beyond Yeonrim — it's deliberately scoped to one venue. But the pattern (small mobile-first platform with admin-approval workflow and OAuth + Firestore real-time state) generalizes to a wider class of small-business event-coordination problems, and a future version could be templated for other small venues running similar event formats.
