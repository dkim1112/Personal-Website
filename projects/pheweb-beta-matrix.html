Overview

Most GWAS analyses look at one trait at a time. This pipeline builds the infrastructure for looking at all of them together — a sign-consistent variant × phenotype matrix and 50-dimensional embeddings ready for downstream representation learning.

The Genome-Wide Association Study (GWAS) field has spent two decades publishing per-trait summary statistics, and several large research consortia (MGI-BioVU, UKB-TOPMed, FinnGen) now publish them through PheWeb — a public viewer that exposes the data through a standard API. The naive way to use that data is one trait at a time; the more interesting question is what happens when you align thousands of traits into a single matrix and look for shared structure. That's a non-trivial engineering problem: each PheWeb instance has hundreds of phenotypes with millions of variants, the allele orientations aren't guaranteed consistent across traits, and the resulting matrix is sparse enough that downstream methods need careful preprocessing. The pipeline handles all of it — parallel downloads with schema detection, allele-orientation correction with sign flipping, p-value filtering, and truncated SVD to produce 50-dimensional variant and phenotype embeddings. It's the data foundation for my ongoing Medical School research and the Dermatology Paradox Atlas proposal I'm working with Dr. Lam Tsoi's group on.

The Problem

Public GWAS data is everywhere. A reusable pipeline for getting it into a form where you can study disease relationships isn't.

Public GWAS summary statistics through PheWeb give the research community access to millions of variant-phenotype associations across multiple large biobanks. But every group that wants to use this data for cross-phenotype analysis ends up writing a one-off script: download N phenotypes, intersect rsIDs, align alleles, build the matrix, hope you got the sign flips right. The result is that the same fragile preprocessing gets rewritten across labs, the same edge cases (palindromic SNPs, duplicate rsIDs, missing p-values, gzip vs. plain TSV) get rediscovered, and the resulting matrices aren't directly comparable between studies. The pipeline this project produces is a step toward fixing that — a single config-driven tool that ingests any PheWeb instance and outputs a sign-consistent matrix plus embeddings, with the limitations of each step documented rather than hidden.

Product Vision

A pipeline a researcher can clone, point at any PheWeb instance, and get back a sign-consistent variant × phenotype β-matrix and its low-dimensional embeddings in one command — with the limitations of each step explicit enough that the matrix can be cited responsibly in downstream analyses.

The matrix and embeddings produced by this pipeline are the data foundation for two ongoing research directions: the UMich Medical School work on multi-phenotype prioritization, and the Dermatology Paradox Atlas — a systematic search for skin-disease pairs whose genetic similarity diverges from epidemiological co-occurrence. Neither of those would be possible if I had to rewrite the preprocessing for every new biobank. Beyond my own use, the pipeline is open-source under Apache-2.0 so other labs can build on the same matrix construction rather than reinventing it.

Core architecture

Four-Stage Pipeline: Download → Build → Embed → Analyze

Stage 1 (1_download_phenotypes.py) hits the PheWeb instance's /api/phenotypes.json endpoint to enumerate all available phenotypes, then downloads summary stats for each in parallel using a 10-worker ThreadPoolExecutor. Schema detection handles the column-naming inconsistencies between PheWeb deployments (rsids vs. rsid; pval vs. p_value vs. p). Stage 2 (2_build_matrix.py) intersects rsIDs across all successfully downloaded phenotypes to construct a dense matrix, then handles the critical allele-orientation step: for each phenotype other than the reference, it compares ref/alt allele orientation against the reference and either keeps β as-is, negates β when alleles are flipped, or sets to NA when alleles don't match either pattern (e.g., palindromic SNPs, multi-allelic variants). A p-value filter at 5e-5 retains only significant associations. Stage 3 (3_create_embeddings.py) applies truncated SVD at k=50 to produce variant and phenotype embeddings. Stage 4 (4_analyze_similarity.py) computes cosine similarity in embedding space and validates against known comorbidity structure.

My Contribution

Sole engineer — designed the four-stage pipeline, implemented the allele-orientation logic, and built it for reuse across PheWeb instances.

This is a one-person research engineering project — no co-author on the code. The most important design decision was treating the pipeline as infrastructure for other research rather than a one-shot script: config-driven for different biobanks, modular stages that can run independently, and a README that documents limitations rather than hides them. Several of those limitations are still open — they're listed in the README and in Lessons Learned below.

What I worked on

  • Parallel data ingestion. Built the PheWeb API and download client with ThreadPoolExecutor for parallelism, gzip auto-detection (some endpoints serve plain TSV, others gzipped), and schema auto-detection across the column-naming variants different PheWeb deployments use.
  • Sign-consistent matrix construction. Implemented the allele-orientation comparison: for each non-reference phenotype, check whether ref/alt matches or is flipped relative to the reference phenotype, then preserve β or negate β accordingly. This is the most subtle step in the pipeline — getting it wrong silently corrupts every downstream analysis. Documented unhandled cases (palindromic SNPs, multi-allelic sites) honestly in the README so users know where the matrix has known limitations.
  • Matrix factorization and embeddings. Wrapped truncated SVD (scikit-learn) at k=50 to produce both variant and phenotype embeddings, with PCA-style variance-explained outputs for sanity-checking the dimensionality choice.
  • Reusability and documentation. Made the pipeline config-driven so swapping PheWeb instances is a one-line edit, licensed under Apache-2.0, and wrote the README to explicitly call out preprocessing limitations rather than implying the matrix is clean. The pipeline currently handles MGI-BioVU; UKB-TOPMed and FinnGen are the next deployment targets.

Lessons Learned

The two unglamorous decisions — how to handle missing data and how to handle allele orientation — decide whether the matrix is research-grade or research-garbage. Most of the engineering effort went into them, and most of the open work still left is in them.

What worked: writing the allele-orientation logic with explicit, named cases (match / flip / mismatch → NA) rather than collapsing them into a single transformation. The first version conflated cases and produced sign errors in ~5% of variants before I rewrote it. Naming each case explicitly made the failure modes legible.

What didn't: the current p-value threshold (5e-5) is hard-coded as a default rather than tested against a sensitivity sweep. I know it matters — different thresholds will produce different embedding structures — but I didn't yet have the apparatus to compare embeddings across thresholds rigorously. The "TBD" annotations still in the README are honest about this gap.

What I'd do differently: I'd build the sensitivity-analysis harness alongside the core pipeline, not after. Specifically, I'd parameterize the threshold and add a comparison step that quantifies how much the downstream embedding cosine-similarity ranking changes when the threshold moves. That's the same principle as building an evaluation harness before tuning a model.

Where It's Going

Three concrete pipeline-hardening priorities are next:

  • NaN-aware matrix factorization. Replace the implicit zero-imputation in SVD with softImpute (Mazumder/Hastie) or iterative SVD with missingness — current zero-imputation conflates "no signal" with "p ≥ threshold," which silently biases the embedding.
  • Palindromic SNP handling. A/T and C/G strand-ambiguous variants are currently dropped to NA when they appear flipped; the cleaner solution is to either exclude them upfront or use allele-frequency matching (the convention used in MungeSumstats and LDpred2).
  • P-value threshold sensitivity sweep. Run the full pipeline at 5e-5, 5e-6, and 5e-8 thresholds and quantify how the downstream embeddings change. Justify whichever threshold makes it into the published matrix.

Three larger extensions are in scope: (1) cross-biobank deployment to UKB-TOPMed and FinnGen for the same dermatology comorbidities, to test whether the embedding structure transfers across populations; (2) integration into the Dermatology Paradox Atlas as the underlying matrix; (3) submission of the pipeline as a methods paper alongside the Atlas, since the reusable matrix-construction tooling is itself a contribution to the PheWAS community.
