Overview

Mapping apps optimize the path between two points. But if you need to visit four stations on the same subway line and return, the order itself affects total time — because direction-specific headway gaps mean the "obviously shorter" geographic ordering can be slower than a longer one that lines up better with the timetable. This tool solves that.

During my military service at Gajeong Station on Incheon Line 2, I regularly had to visit multiple stations along the line and return — a task none of the major mapping platforms (Naver Map, Kakao Map) handle, because they optimize point-to-point routes, not multi-stop sequences with return. The interesting observation that kept coming up was that the order of stops, not just the choice of stops, materially changed total time. Visiting station B before C might leave you waiting eight minutes for the train back; visiting C before B might give you a two-minute wait. I built this tool to model the real Incheon Line 2 timetable, search all visit orderings exhaustively, and return the one with minimum total time — what I call the Wait-Time Paradox: shorter geographic distance isn't shorter total time when headway gaps are large.

The Problem

Multi-stop transit is missing from mapping platforms — and on a single subway line, visit order alone can swing total time by 10+ minutes.

The standard mapping-app abstraction is "shortest path between A and B." But the actual operational task — "visit A, B, C, and D, then return to start" — is a different problem, and one that nobody on Naver, Kakao, or Google Maps lets you solve. On a single subway line, the problem is sharper than it first appears: train headways aren't symmetric in both directions, off-peak gaps can stretch to 9 minutes, and peak frequencies drop to 3 minutes. That asymmetry means the optimal visit order depends on when you arrive, not just where you're going. A naive "nearest-neighbor" sequence routinely loses to an ordering that geographically backtracks but waits less.

Core Architecture

Time-Dependent Traveling Salesperson Problem (TD-TSP) with Brute-Force Search Over Real Timetable Data

The problem is a TD-TSP: minimize total time across a closed tour where each edge cost is the sum of (wait for next train) + (travel time to next stop) + (dwell time at next stop), and the wait component depends on arrival time at the previous stop. For the realistic operational case — up to ~6 stations to visit in one trip — brute-force enumeration of all permutations is computationally trivial (≤ 720 paths) and guarantees the optimum, which heuristics don't. Each candidate path is evaluated against the actual Incheon Line 2 timetable: at every stop transition, look up the next train in the relevant direction, compute the wait (with a 2-minute buffer for boarding), add travel time, add dwell time. Total time per path = sum of these across the closed tour. The best path is returned along with a breakdown of where time was spent.

My Contribution

Sole engineer — formulated the optimization problem, built the timetable model, implemented the search, and shipped the deployed UI.

This was a one-person project, scoped to a problem I was actually living with at the time. The most useful design decision was matching the algorithm to the real input size. TD-TSP is NP-hard in general and there's a deep literature of approximation algorithms — but my actual use case was 4–6 stations per trip, where brute force is both faster to implement and provably optimal, while a heuristic would have been more code, fewer guarantees, and pointless overhead at this scale.

What I worked on

  • Problem formulation. Defined the total-time objective as wait + travel + dwell summed over the closed tour, identified the time-dependent edge weights as the key difference from standard TSP, and picked brute force as the appropriate algorithm for the realistic input size.
  • Timetable modeling. Built the data layer around the actual Incheon Line 2 schedule, including both-direction headways and edge cases (midnight crossings, where consecutive trains span the date boundary).
  • Search engine. TypeScript implementation that enumerates permutations, evaluates each against the timetable, and returns the optimal ordering with a per-stop time breakdown.
  • UI. Responsive single-page interface that lets the user pick origin and visit set, displays the optimized order with arrival/departure times for each leg, and surfaces the Wait-Time Paradox visually (showing how a different order would have changed total time).

Key Achievements

A working, deployed tool that demonstrably out-performs sequential ordering on the line I actually used it on.

The Wait-Time Paradox isn't hypothetical — for real trips I needed to take during my service, the optimizer returned visit orders that were meaningfully faster than the nearest-neighbor sequence I'd have picked by intuition. The tool is deployed and usable today; it covers Incheon Line 2 specifically because that's the line I had the use case for, not because the algorithm is line-specific.

Lessons Learned

Matching algorithm to actual problem size matters more than choosing the most sophisticated algorithm. Brute force on 720 permutations gives provable optimality in milliseconds; a heuristic would have been more code, fewer guarantees, and no real speed benefit at this scale.

What worked: writing out the time-dependent cost function explicitly before picking an algorithm. The TD-TSP framing made it obvious that ordering mattered (the cost depends on arrival time, not just adjacency), and once that was clear, the brute-force decision was easy to justify.

What didn't: I initially modeled the timetable as a list of trains rather than as a sorted lookup structure. For a 6-station tour with 720 permutations × multiple stop transitions, the cost of unsorted lookup added up. Switching to a sorted-by-time representation with binary-search lookup was a small refactor that made the search noticeably snappier.

What I'd do differently: I'd build a small comparison view from day one showing "your route under naive ordering" vs. "optimized order" with the time difference. That's what makes the Wait-Time Paradox visible to users; without it, the tool just shows an answer without showing why the answer is non-obvious.

Where It's Going

The single concrete extension is integrating real-time train status — the current version uses the published static timetable, which is accurate but doesn't account for delays. The Incheon Transit Corporation's open data API would close that gap.

The deeper extension is generalizing beyond a single line. Multi-line routing introduces transfers, which significantly complicates the cost function (transfer wait + walking time + line-switching delay), and the input size grows enough that brute force breaks down — at that point, a real TD-TSP heuristic (or a constraint solver) becomes necessary. This was deliberately scoped to single-line to keep the search exact; the multi-line version is a different project.
