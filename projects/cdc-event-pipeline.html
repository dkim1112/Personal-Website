Overview

Most data pipelines poll on a schedule. This one reads PostgreSQL's Write-Ahead Log directly, capturing every row-level change at the moment it happens — so the full lifecycle of an order (created → paid → shipped → delivered) survives instead of collapsing into whatever state happened to be there when the poll ran.

A scheduled pipeline that SELECTs every 10 minutes has three structural blind spots: intermediate states vanish between polls (CREATED → PAID → SHIPPED in 8 minutes shows up only as SHIPPED), deletes are invisible (you can't see a row that's no longer there), and source-DB load is paid every cycle even when nothing changed. Change Data Capture solves all three by reading the database's own log instead of polling it. This pipeline implements the full CDC pattern end-to-end: PostgreSQL with logical replication, Debezium as the connector reading the WAL, Apache Kafka as the durable event stream, and a Python consumer that writes both an append-only event log and materialized snapshots of current state. It's built around a simulated e-commerce dataset (orders, products, inventory, customers), runs locally via Docker Compose, and includes a 5-page Streamlit dashboard for visualizing event flow, processing lag, and reconstructed snapshots. The repo includes 8 documented Architecture Decision Records, 21 unit tests, and an AWS deployment cost analysis (~$83/month).

The Problem

Polling-based pipelines silently lose information every cycle. CDC is the right tool — but the right tool is rarely the easiest one to set up.

The intermediate-state problem is the clearest motivator. If an order transitions CREATED → PAID → SHIPPED within a single polling interval, every downstream analysis sees only SHIPPED. The full lifecycle — how long payment took, which orders stalled at PAID, what percentage were cancelled before shipping — is invisible. That's not just analytical loss; it's the difference between knowing your funnel and guessing at it. CDC fixes the problem at the source, but setting it up correctly is non-trivial: Debezium runs inside Kafka Connect (not standalone), decimal columns come through as base64 by default, the source DB's WAL can bloat if the consumer stops, and out-of-order events show up even on a single Kafka partition because Debezium batches transactions. This project's value isn't that CDC is novel — it's that it's the correctly-set-up version with the production gotchas already discovered, documented, and worked around.

Product Vision

A reference implementation of CDC that anyone can clone, start with one command, and use to understand both the pattern (WAL → Kafka → consumer → snapshots) and the production realities (where exactly each gotcha lives and how to handle it).

The pipeline is built around a simulated e-commerce workload — order lifecycles with 10% cancellations, 5% refunds, realistic timing — because the value is in seeing the CDC pattern work end-to-end, not in the specific business data. The 5-page Streamlit dashboard surfaces every piece of the system: per-event processing lag, the lifecycle transitions reconstructed from raw events, current snapshot state derived from those transitions, dead-letter events that failed processing, and pipeline-ops monitoring. The entire architecture comes up clean from a single docker compose up and is verifiable end-to-end with a consistency-check script that compares snapshots back against the source DB.

Core architecture

WAL-Sourced CDC with Append-Only Event Log and Materialized Snapshots

PostgreSQL is configured with logical replication and REPLICA IDENTITY FULL on all source tables so every row change emits the full before/after state to the WAL. Debezium (running inside Kafka Connect) reads the WAL and publishes JSON change events to five Kafka topics — one per source table. A Python consumer (confluent-kafka) reads these events and does three things: (1) writes them to an append-only event_log table with UUID5-based deduplication (event ID = hash of table + LSN + txId + key, so the same event always gets the same ID, and a re-delivered event is idempotently skipped), (2) updates materialized order_snapshots and inventory_snapshots tables that represent current state derived from the event stream, and (3) records per-minute processing metrics for monitoring. Failed events land in a dead_letter_events table rather than being dropped. The snapshots are derived from events rather than copied from the source, which means they can be rebuilt from the event log at any time — that's what makes this event sourcing, not just CDC. Eight Architecture Decision Records document the major design choices in 02-DECISIONS.md.

My Contribution

Sole engineer — designed the architecture, navigated the production gotchas, and built the demo end-to-end with the same documentation discipline as a small production service.

This is one of two streaming/batch sister projects I built to develop the data engineering muscles needed for production ETL work — the batch counterpart is the Korean Price Monitor pipeline. CDC was the harder of the two to set up correctly, less because the conceptual pattern is complex (WAL → Kafka → consumer is a few words) and more because the productionization details (Debezium's connector lifecycle, decimal encoding, WAL bloat under consumer stalls, out-of-order events) are the kind of thing you discover by hitting them rather than by reading docs. Documenting each as it came up, in 02-JOURNAL.md and the 8 ADRs, was the way I made sure I'd actually internalized the lessons rather than just gotten the pipeline to "work."

What I worked on

  • CDC pipeline architecture. Designed the WAL → Debezium → Kafka → consumer → analytics-DB flow. Made the deliberate decision to derive snapshots from events (event sourcing) rather than mirror the source DB, so the analytics layer can always be rebuilt from the event log.
  • Idempotent event processing. Implemented UUID5-based deduplication keyed on (table, LSN, txId, primary key), so a re-delivered event always gets the same ID and is silently skipped on second insert. Verified zero duplicates across a 533-event burst test.
  • Failure-mode handling. Built the dead-letter queue for failed events (rather than dropping or crashing the consumer), per-table out-of-order detection using event timestamps, and end-to-end lag monitoring (event-creation-time vs. event-processing-time per minute). 21 unit + integration tests cover the core processing logic.
  • Operability. Built the 5-page Streamlit dashboard, a CLI monitor (monitor.py --watch), a consistency-verification script (verify_consistency.sh) that compares snapshots against the source DB, and the Docker Compose stack with 5 services that come up clean from one command.

Key Achievements

An end-to-end CDC reference implementation that captures every state transition, handles real production gotchas, and runs reproducibly from a single command.

The dashboard's Order Snapshots page is the clearest demonstration of why CDC matters: from raw row-level events alone, the pipeline reconstructs the full order-lifecycle flow — 65 CREATED events, 60 CREATED→PAID transitions, 48 SHIPPED→DELIVERED transitions, 5 CREATED→CANCELLED — none of which would be visible in a polling pipeline that sampled every 10 minutes. The pipeline survived a 533-event burst test with zero duplicates (UUID5 dedup working as designed) and out-of-order events were caught by per-table timestamp tracking. The repo's 8 ADRs, 21 tests, schema design doc, decision journal, and AWS cost breakdown all exist because the goal wasn't just to build a working pipeline — it was to leave a complete enough record that someone else (or future me) could understand why each choice was made, not just what it was.

Lessons Learned

CDC's value proposition is easy to state and surprisingly hard to set up. The gap between "I understand the pattern" and "I have a working CDC pipeline with production-correct behavior" is mostly populated by gotchas that aren't in the docs.

What worked: writing an ADR for each major decision as I made it, not retroactively. ADRs written after the fact tend to read like marketing for a choice that's already been made; ADRs written in the moment of the decision force you to articulate the tradeoff you're weighing — and to come back later and notice when the tradeoff changed.

What didn't: I underestimated how many of the gotchas were configuration details rather than design problems. The Debezium decimal-encoding issue (base64 by default, needs decimal.handling.mode: "string") is a one-line config change — but I lost most of a day to it because the failure mode (silently mangled financial values) doesn't look like a config issue from the symptoms. The general lesson: when something in a managed connector is producing weirdly-shaped data, check the connector's config docs before writing custom transformation code.

What I'd do differently: I'd write a "first principles sanity check" step into the pipeline from day one — a script that compares raw connector output against the source-DB ground truth, flagging any field where the round-trip doesn't match. The decimal-encoding bug, the out-of-order event issue, and a couple of other small ones would have been caught by this in minutes rather than hours. That's the streaming-equivalent of the data-quality validation pattern from the Price Monitor pipeline.

Where It's Going

Three concrete extensions are documented in 02-ROADMAP.md. The first is partitioned consumption with consumer groups so the pipeline can scale beyond demo throughput (~50 events/min) — this is the single biggest gap between the current design and a real production system. The second is schema registry integration (Confluent Schema Registry with Avro or Protobuf) so events get validated at ingest rather than relying on JSONB flexibility. The third is KRaft mode to drop the Zookeeper dependency (Kafka 3.x+).

The deeper extension is connecting this CDC pipeline to the Price Monitor pipeline as two sides of the same data-engineering pattern: real-time event sourcing for transactional workloads, batch ETL with quality validation for analytical workloads. They share the same documentation discipline (ADRs, schema design docs, journals, cost analysis), and the goal is to develop a coherent two-project answer to "can you build the data infrastructure layer?" for any future DE/SWE role I'm applying to.
