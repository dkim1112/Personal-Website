Overview

Two government APIs publish Korean consumer price data in different formats at different frequencies. This pipeline joins them, validates the data quality of every run, and answers a single question: when official CPI says inflation is X%, is that what consumers are actually paying?

I noticed grocery bills weren't tracking with the official inflation numbers and wanted to know whether my impression was actually true at scale. Statistics Korea (KOSTAT) publishes product-level e-commerce prices weekly in XML — about 600K product listings per collection across 124 item categories. The Bank of Korea (ECOS) publishes monthly CPI indices in JSON, with 1,743 hierarchical items keyed to a base year of 2020. They share classification codes, which means the comparison is possible — but neither source produces a joined view, neither is robust against the other's schema drift, and neither validates its own freshness or completeness. The pipeline I built handles all of that: defensive API integration, raw-data preservation, six data quality checks with thresholds calibrated against real distributions rather than textbook defaults, and a four-page Streamlit dashboard that produces the "Price vs. CPI" view that started the question in the first place.

The Problem

The official inflation index and the prices in front of consumers come from the same government but don't talk to each other.

Korean consumer price data is published by two agencies in incompatible formats. KOSTAT publishes the granular reality — actual e-commerce prices, product by product, ~600K listings per collection cycle — in XML, weekly, with a ~2-week lag. ECOS publishes the official inflation summary — CPI indices, 1,743 categories, base year 2020 — in JSON, monthly. The two sources share classification codes, so the comparison is technically possible: for any given category, you could check whether product prices are tracking the official CPI for that category or drifting from it. But nobody had built that joined view, and crucially, nobody was validating that either source had returned good data on any given run. KOSTAT in particular has undocumented API behaviors — successful responses omit the result code that the documentation says is always present, item codes are 6 characters where docs say 7, etc. — which means a naive pipeline silently produces garbage on a bad day.

Product Vision

A locally runnable pipeline that any analyst, journalist, or curious consumer could clone, point at the two government APIs, and use to ask: in the categories I actually buy, is the official inflation index tracking reality?

The dashboard's "Price vs. CPI" page is the answer to the original question. For each of 124 mapped categories, it overlays median product price (KRW) against the CPI index for that category. When the lines diverge — product prices rising faster than the CPI for the same category — that's an empirically visible gap between official inflation and what consumers in that category are paying. Below the chart, an inline callout calculates the divergence: "Product prices changed +X% while CPI changed +Y%." The data-quality and pipeline-ops pages exist because that comparison is only worth something if the underlying collections are themselves trustworthy.

Core architecture

Raw-Preserving ETL with First-Class Data Quality and Schema Drift Detection

Two collectors hit the source APIs on independent schedules — KOSTAT weekly via cron, ECOS monthly. Every run begins with a schema drift check that compares the current response shape against a baseline; deviations flag for review before they corrupt the raw store. All raw data lands unaggregated in PostgreSQL (raw schema, monthly-partitioned for the high-volume KOSTAT table, with UPSERT deduplication for ECOS). A separate aggregation stage runs SQL-based raw-to-mart transformations: mart.daily_price_summary with median and IQR, mart.monthly_cpi_index with latest-wins semantics, and mart.price_vs_cpi as the joined analytical view. After aggregation, six quality checks run against the mart, with thresholds calibrated against observed distributions rather than textbook defaults — for example, the CPI range check was initially set to [80, 130] based on "CPI ≈ 100" and produced 143 false positives in the first run; the corrected range of [30, 250] reflects how widely real sub-indices actually vary. Eight Architecture Decision Records document the rationale behind each major choice.

My Contribution

Sole engineer — designed the architecture, wrote every component, calibrated the quality thresholds against real data, and documented the decisions.

The work was solo from first commit to final dashboard. The most useful discipline I imposed on myself was writing an Architecture Decision Record (ADR) for each significant choice — pivoting away from KAMIS, choosing Postgres over DuckDB, storing raw and aggregating later, choosing cron over Airflow, calibrating thresholds against data. The ADRs are short, but they meant that every time I considered changing something later, I had to engage with the prior reasoning rather than just override it. That habit kept the pipeline more coherent than a single-author project usually ends up.

What I worked on

  • Defensive API integration. Built KOSTAT and ECOS collectors with adaptive date probing, retry logic, and discovery-driven parsing — necessary because KOSTAT's documented behavior diverges from actual API behavior in multiple places (item code length, presence of result codes on success).
  • Schema drift detection and data quality. Designed the six-check validation suite (freshness, completeness, null ratio, anomaly detection, CPI range, schema baseline) and the workflow that calibrates each threshold against real observed distributions rather than guessing. Recalibrated the CPI range check after the first deployment produced 143 false positives.
  • Raw-preserving database design. Schema design in 01-SCHEMA-DESIGN.md: monthly-partitioned raw tables for KOSTAT's high-volume product listings, deduplication-indexed raw tables for ECOS, and SQL-based mart aggregation as a separate stage so the raw data is always preserved and re-aggregatable.
  • Dashboard, monitoring, and operability. Built the 4-page Streamlit dashboard (Price Trends, Price vs. CPI, Data Quality Health, Pipeline Ops), the cron scheduling layer, Slack webhook alerting, and 23 unit + integration tests.

Key Achievements

An end-to-end pipeline that answers the original question — do product prices track the CPI? — and is honest about its own data quality at every stage.

The dashboard's Price vs. CPI page shows the empirical gap I started with: for several mapped categories, product prices have shifted meaningfully against the CPI index across the observation window. The findings are descriptive — this is a single-cohort cross-source comparison, not a causal claim about what's wrong with the CPI — but they're the kind of finding that would have been invisible without joining the two government datasets. Beyond the analytical output, the more durable contribution is the engineering pattern itself: a small, well-documented data pipeline where every threshold is justified, every architectural decision is on the record, and the data quality story is told to the operator as plainly as the analytical story is told to the reader. The whole system is 23 tests, 8 ADRs, and a Docker Compose stack that comes up clean from a clone.

Lessons Learned

Thresholds need data, not theory. Every quality check I tuned by intuition first and then by observation produced false positives until I rebuilt the threshold against the actual distribution. Looking at the data before choosing the rule is the unglamorous step that decides whether the rule helps or hurts.

What worked: the discipline of writing an ADR for each major decision. It's overkill on a solo project, technically — but it forced me to engage with the why of each choice rather than the what, and it meant that later changes were grounded in prior reasoning rather than reflex.

What didn't: I assumed government API documentation could be trusted. The KOSTAT API in particular diverges from its docs in ways that fail silently — successful responses omit the result code, item codes are 6 characters where docs say 7. A whole class of early bugs came from believing the docs over what the API actually returned. The lesson: a single hour spent inspecting actual API responses before writing the collector would have saved several days of subsequent debugging.

What I'd do differently: I'd start with a small sample of real API responses, not the docs. Specifically, I'd build a "what does this endpoint actually return?" diagnostic step as the very first piece of the pipeline, before any business logic. That diagnostic would also be the natural place to seed the schema drift detection baseline.

Where It's Going

Two well-scoped extensions are documented in 01-ROADMAP.md. The first is PPI (producer price index) integration from ECOS stat codes 404Y014–404Y017, which would let the dashboard tell a more complete inflation story — consumer-facing prices vs. wholesale/producer prices vs. official CPI. The second is moving the dashboard from single-machine local Streamlit toward something a small audience could use (authentication, a hosted deployment, cost-analyzed against AWS at ~$35–50/month per the README's existing breakdown).

The deeper question this project hints at is whether the analytical pattern generalizes — does the same "raw-preserving, quality-validated, schema-drift-aware" pipeline structure apply to other public Korean data sources where the comparison-of-records-vs-aggregates problem exists? Labor statistics, housing data, tax revenue. The price-monitoring pipeline is interesting in itself, but it was also a deliberate exercise in building a template I could re-apply.
